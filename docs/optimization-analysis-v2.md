# 第二轮优化分析：体验 + 功能客观评估

> 2026-06-29 — 抛开"别人怎么做"，纯从真实用户视角审查每个操作环节

---

## 评估方法

模拟用户完整工作流，逐环节记录摩擦点：

```
启动 → 看数据 → 搜索筛选 → 爬新数据 → 看详情 → 生成标书 → (AI分析) → 退出
```

每处按严重度打分：**🔴 堵点（必须修）| 🟡 痛点（该修）| ⚪ 痒点（可修）**

---

## 一、启动环节

### 🔴 首次启动无视觉反馈

点 `desktop.py` 后要等 jieba 加载模型（~0.5s）+ 数据库初始化（首次 ~2s）。这期间窗口不出来，用户不知道发生了什么。

**建议**: `main()` 里先弹 splash 或至少 print "正在启动..."。

### ⚪ jieba 每次都打印 Loading 日志

```
Building prefix dict from the default dictionary ...
Loading model from cache /tmp/jieba.cache
Loading model cost 0.533 seconds.
```

终端能看到，GUI 看不到。不影响功能，但首次加载的 0.5s 延迟让窗口弹出来时表格已经填好了——实际上体验不算差。

---

## 二、搜索筛选环节

### 🟡 搜索词 + 筛选条件是 AND 还是 OR，用户不知道

当前逻辑：搜索词走 FTS5 MATCH，筛选走内存过滤，两者是 **AND** 关系。这是合理的，但状态栏没指示——"你在看筛选后的 9 条"能看到，但"这 9 条是因为搜了什么 + 筛了什么"看不到。

**建议**: 状态栏已有"显示：9/36条"，够用了。不再加。

### 🟡 日期筛选无"近7天"快捷按钮

有"今天"和"本周"但没有"近7天"。数据按天更新的时候最常查的就是最近一周。

**建议**: 加"近7天"按钮，一行代码。

### 🟡 筛选条件保存不下来

关闭窗口再打开，筛选条件全回默认。每次都要重新设地区/类型/日期。

**建议**: 退出时保存筛选条件到文件，启动时恢复。或者不做（大部分桌面工具本来就不保存筛选状态）。

---

## 三、爬取环节

### 🔴 爬虫失败只有日志文字，没有状态栏提示

网络不通或 ggzy 挂了，日志窗口里显示 `[X] 爬取失败`，但日志窗关了之后用户不知道爬没爬成功。状态栏没变。

**建议**: 爬取结束后状态栏短暂显示"上次爬取: 成功 20条 / 失败"。

### 🟡 不能中途取消爬取

日志窗口没取消按钮。点了"爬取最新"后要等到 20 条全抓完（约 40 秒）才能操作。

**建议**: 加取消按钮，设 `_cancel_flag`，循环里检查。

### 🟡 爬取关键词输入框没有 placeholder

输入框是空的，用户不知道填什么。要加 `placeholder="留空=全量，如：停车场"` 提示。

tkinter Entry 不支持原生 placeholder，但可以设灰色默认文字。

---

## 四、详情查看环节

### 🟡 详情弹窗不能复制文本

`state=tk.DISABLED` 后用户无法选中文字。想复制项目编号或标题要手动打。

**建议**: 改成 `state=tk.NORMAL` + 绑定 `<Control-c>` 或者保持 DISABLED 但列表里加复制按钮。

### 🟡 详情弹窗没有"原文链接"可点击

数据库有 `url` 字段（每条公告的 ggzy 原链接），详情弹窗里没显示。用户想看原文要手动拼 URL。

**建议**: 详情弹窗加一行"原文链接: ..."。

---

## 五、标书生成环节

### 🔴 知识库占位符 `XXXXXX` 被 AI 如实输出

`knowledge/company.yml` 里 `number: "D251XXXXXX"` → AI 模式会把 `D251XXXXXX` 当成真实资质编号写入标书。**这是法律风险。**

**建议**: `load_knowledge_context()` 检测占位符，检测到后：1) 在上下文中追加警告文字 2) AI 模式生成时提示用户先填知识库。

### 🟡 标书预览是 Markdown 原文

模板模式生成的标书是 Markdown 格式（`# 标题`、`**粗体**`、`> 引用`），预览区显示原始 Markdown 语法，不渲染。对没写过 Markdown 的用户不友好。

**建议**: 两个方向选一：(A) 预览区做简单渲染（标题加粗变色）(B) 直接生成纯文本而非 Markdown。当前 Markdown 格式对程序员友好，保持即可。

### 🟡 标书导出位置不明确

`_export_bid()` 写到当前工作目录，文件名 `投标书_20260629_230000.md`。用户可能不知道文件在哪。

**建议**: 弹出文件保存对话框 `filedialog.asksaveasfilename()`，让用户选位置和文件名。

---

## 六、AI 分析环节

### 🟡 没 API Key 时的错误信息不够友好

`ai_analyzer.py` 调用失败时返回 `[X] AI 分析失败：...` 后面跟 OpenAI SDK 的原始报错（SSL 错误、401 认证失败等）。普通用户看不懂。

**建议**: 检测常见错误码，翻译成中文提示：
- `401` → "API Key 无效，请确认 config.py 中 DEEPSEEK_API_KEY 已配置"
- 网络错误 → "无法连接 DeepSeek API，请检查网络"
- 其他 → 显示原始错误

### 🟡 AI 功能全部走线程但无超时

DeepSeek API 可能响应很慢（> 30 秒），线程一直等着。用户以为卡死了。

**建议**: `_run_ai_task` 加超时提示——10 秒没返回就在输出区显示"响应较慢，请耐心等待..."。

## 七、退出环节

### ⚪ 关闭窗口没确认

直接点 X 就关了。因为所有数据都在 SQLite 里，即使没保存也不丢数。不太需要确认。

## 八、功能层面

### 🔴 无 CSV 导出

当前能看不能出。筛选出 5 条四川可投商机后，想发给同事只能截图。

**建议**: 加"导出 CSV"按钮，当前表格可见列 + 隐藏的 URL 也导出。15 行代码。

### 🟡 无法在浏览器打开原文

每条公告的 `url` 字段存着 ggzy 原文链接。但 GUI 里完全没暴露。用户想看详情只能复制标题去百度搜。

**建议**: 详情弹窗加"在浏览器打开"按钮 → `webbrowser.open(url)`。

### ⚪ CLI 的 crawl 跟不上 GUI

`main.py crawl` 走 `source_manager.crawl_all_enabled()`，能多源。但之前 `crawl` 命令有单独调 GgzyCrawler 的老代码已经删了——这个是对的，CLI 和 GUI 走同一条路径了。

### 🟡 主表 title/content 存的是 jieba 分词结果

入库时 `" ".join(jieba.cut(title))`，展示时 `.replace(" ", "")` 还原。中间如果碰到英文单词（EPC、POSM）被拆成 E P C 就没法还原。

**长期风险**: 如果未来要对标题做二次分析（如按关键词分类），分词后的存储会干扰。

**当前影响**: 极小。展示正常、搜索正常。

## 十、不做（重申）

| 项目 | 原因 |
|------|------|
| 统计图表 | 36条数据画图没意义 |
| 筛选条件持久化 | 桌面工具通常不保存筛选状态 |
| 窗口关闭确认 | SQLite 自动持久化，不丢数据 |

---

## 小结

**最该做的 3 件**：

1. **知识库占位符校验** — 防 AI 输出 `XXXXXX`，安全底线
2. **CSV 导出** — 从"只能看"变成"能用"
3. **爬虫失败状态反馈** — 让用户知道爬没爬成功

做完这三样，工具就闭环了。其余 7 项都是体验打磨，不阻塞使用。
