## 二、架构层问题

### A1. desktop.py 太胖 (759行, 45个方法) — 严重度: 中

三Tab + 源管理 + 爬取 + URL抓取全挤在 `BidToolApp` 一个类里。改一个 Tab 容易踩到另一个。

**建议**: 拆成 `desktop/tab_opportunity.py`, `desktop/tab_bid.py`, `desktop/tab_ai.py`，每个 150-200 行。`BidToolApp` 只做初始化组装。

### A2. main.py 内嵌 FastAPI Web (200行) — 严重度: 中

`cmd_web()` 里有 150 行 HTML + 3 个路由定义。跟 CLI 入口混在一起。

**建议**: 拆到 `web/app.py` + `web/templates/index.html`。

### A3. 内部懒加载过多 — 严重度: 轻

`desktop.py` 有 8 处在方法内 `from xx import yy`。有些是必要的（避免循环引用），有些不是（`from db import get_db`）。

**建议**: 把非循环引用的内部 import 提到文件顶部。

### A4. 入口分散 — 严重度: 轻

`desktop.py` 和 `main.py gui` 两处都能启动 GUI。没有统一入口点。

**建议**: 统一为 `python -m bid_tool` 或明确 `python desktop.py` 是主入口。

## 三、功能缺口

### B1. 无法删除已入库数据 — 严重度: 高

爬回来的垃圾公告（"绿地养护""排污权交易"等与停车无关的）没法在 GUI 里删。只能手动写 SQL。

**影响**: 数据库积累噪音，筛选结果越来越脏。

**建议**: 加删除功能 — 选中行 → 删除按钮或键盘 Delete 键。改动量约 20 行。

### B2. 无法导出 — 严重度: 中

筛选后只能看，不能导出 CSV/Excel。没法做报表、发邮件、打印。

**建议**: 加「导出 CSV」按钮，把当前表格内容写出 `.csv`。改动量约 15 行。

### B3. 数据可视化缺失 — 严重度: 中

`report.py` 有文本报告，也有 `report_dict()` 返回结构化数据。但 GUI 里没有任何图表——地区分布、类型占比、评分分布全是数据不够直观。

**对标**: 商业产品有 ECharts 图表、GIS 热力图、趋势折线图。

**建议**: 基于已有 `report_dict()` 数据，在 GUI 加一个简单的统计面板——柱状图（地区分布）、饼图（类型占比）。用 tkinter Canvas 画，或者 matplotlib 嵌入。改动量约 80 行。

### B4. 没有通知推送 — 严重度: 轻

高优新商机入库后没有主动推送。

**建议**: 远期做。加 webhook（企业微信/钉钉）或邮件。当前可先加桌面通知（`tkinter.messagebox` + 声音）。

### B5. 爬虫无重试 — 严重度: 中

`BaseCrawler._fetch_urls()` 里只有一次 `requests.get()`，网络抖动就丢数据。

**建议**: 加 `retry` 机制（3 次 + 指数退避）。改动量约 10 行。

### B6. 爬虫关键词过滤在详情之后才做 — 严重度: 轻

当前逻辑：取列表 → 爬详情页 → 判断标题含关键词。关键词不匹配的话详情页白爬了。

**建议**: 如果列表层就能拿到标题（ggzy 可以），先在列表层过滤。只在没办法的时候才先爬后筛。

## 四、代码健壮性

### C1. 异常处理不统一 — 严重度: 中

- `desktop.py` / `ai_analyzer.py` / `bid_writer.py` 有 6+ 处 `except Exception`
- `db.py` / `search.py` / `extractor.py` 完全没有异常处理
- 有些模块吞异常（`except: pass`），有些模块让异常上抛

**建议**: 统一策略——数据层不吞异常，UI 层 catch + 提示。

### C2. SQLite 连接管理不一致 — 严重度: 轻

`get_db()` 返回连接，依赖调用者手动 `conn.close()`。`search.py` 和 `report.py` 每次都关，但如果有异常就漏关了。

**建议**: 用 `contextlib.contextmanager` 或 `with` 语句。

### C3. 知识库占位符未校验 — 严重度: 中

`knowledge/company.yml` 和 `team.yml` 里资质编号全是 `XXXXXX`。AI 标书模式会原文输出这些占位符，用户可能不知道。

**建议**: `load_knowledge_context()` 加校验——检测 `XXXX` 占位符，发现后在上下文末尾附加警告。改动量约 15 行。

### C4. AI 上下文可能超限 — 严重度: 中

`ai_analyzer.ask()` 一次性把 50 条商机全文塞进 prompt。数据涨到 500 条后直接超 DeepSeek 64K 限制。

**建议**: 加截断保护——按 token 估算，超过 32K 就只取前 N 条。

### C5. 无测试 — 严重度: 中

所有验证靠临时 `python3 << 'PYEOF'` 脚本。功能多了之后回归成本高。

**建议**: 建 `tests/` 目录，最少把 `tagger`、`search`、`extractor` 三个核心模块加上测试。

## 五、数据层

### D1. FTS5 搜索安全 — 良好

MATCH 查询全部使用参数化 `?` 占位符，无 SQL 注入风险。

### D2. 标题/正文存的是 jieba 分词后的空格串 — 需注意

入库时做了 `" ".join(jieba.cut(title))`，展示时 `title.replace(" ", "")`。展示没问题，但数据库里的 title 不是原文。未来如果要做"复制原文标题"或"重新处理"会有偏差。

**建议**: 主表存原文，FTS 索引存分词结果。但当前做法参照 gxcg-spider，社区验证可行，暂不改。

### D3. _normalize_region 位置 — 良好

映射表定义在 `tagger.py`（权威位置），`crawlers/ggzy.py` 从 tagger import。之前已经消除重复定义。

## 六、用户体验

### E1. 详情弹窗只读 — 轻

`_show_detail()` 的 ScrolledText 设了 `state=DISABLED`，不能复制文本。实际场景下用户可能想复制标题或正文。

**建议**: 设成 `tk.NORMAL` 但不可编辑，或者加复制按钮。

### E2. 窗口关闭无确认 — 轻

直接点 X 关闭窗口，没有"确定退出？"确认。

**建议**: 绑 `WM_DELETE_WINDOW` 协议。

### E3. 标书预览字体 — 轻

当前用 `("monospace", 9)`，中文显示可能不美观。

### E4. 状态栏不显示最近更新时间 — 轻

不知道数据是什么时候更新的。

**建议**: 加「最近更新: 2026-06-29 23:00」。

## 九、总结

**当前核心闭环已经完整** —— 爬数据 → 入库 → 搜索筛选 → 标书生成。功能层面最缺的三样：

1. **删除**（数据多了需要清理）
2. **统计图表**（让数据会说话）
3. **知识库校验**（防止占位符被 AI 输出）

代码层面最值得做的是 **异常处理统一** 和 **main.py Web 拆分**。

做完 P0 三项，工具就从"能用"升级到"好用"。
